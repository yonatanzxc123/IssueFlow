# IssueFlow - Implementation Notes

## 1. Project Overview

IssueFlow is a Java 21 Spring Boot backend for the AT&T TDP ticket-management assignment.

The application is built with:

- Java 21
- Spring Boot 3
- Spring Security with JWT authentication
- Spring Data JPA / Hibernate
- PostgreSQL for local runtime persistence
- Maven Wrapper for build and test commands
- Docker Compose for local PostgreSQL setup

For exact setup, build, run, and test commands, see `run.md`.

## 2. Implemented Features

The project implements the major README/PDF requirements:

- Users API
- JWT Authentication API
- Logout with persisted revoked-token support
- Projects API
- Tickets API
- Soft delete and restore for projects and tickets
- Comments API
- `@username` mentions in comments
- Ticket dependencies and blockers
- Preventing `DONE` transition while blockers are unresolved
- Auto-assignment and project workload
- CSV ticket import/export
- Attachment upload, list, and delete
- Automatic overdue ticket escalation scheduler
- Read-only Audit Log API
- Global validation and error handling
- Automated integration/regression tests
- Manual Postman validation artifacts

## 3. Authentication and Security

All endpoints are JWT-protected except:

- `POST /users`
- `POST /auth/login`

Protected endpoints require:

```http
Authorization: Bearer <token>
```

Logout is implemented by persisting the token in a revoked-token deny-list. Once logged out, the same token is rejected by protected endpoints.

Password hashes are never returned in API responses.

## 4. Implementation Note: User Password Compatibility

The README contract for `POST /users` does not include a password field, while `POST /auth/login` requires a username and password.

To preserve compatibility with the README API contract:

- `POST /users` accepts the README body without a password.
- `POST /users` also supports an optional `password` field.
- If no password is provided, the default local/development password is `secret`.
- Passwords are always stored as BCrypt hashes.
- Password hashes are never exposed through DTO responses.

This default password behavior exists for assignment and local testing compatibility only. It should not be treated as a production password policy.

## 5. Auto-Assignment Interpretation

The README/PDF describe auto-assignment to the least-loaded `DEVELOPER` in the project, but no `ProjectMember` model or API is defined.

This implementation therefore uses the following interpretation:

- All users with role `DEVELOPER` are candidates.
- Users with role `ADMIN` are excluded.
- Workload is counted per project.
- Workload includes non-`DONE`, non-deleted tickets assigned to that user in the target project.
- Ties are broken by the lowest user id, which corresponds to the oldest registered user.
- No `ProjectMember` table, model, or API was added.

## 6. Attachment Storage

Attachments are stored in PostgreSQL using the existing attachment binary content field.

Attachment API responses return metadata only:

- `id`
- `ticketId`
- `filename`
- `contentType`
- `sizeBytes`

Raw file content is not returned in normal JSON responses.

Allowed attachment content types are:

- `image/png`
- `image/jpeg`
- `application/pdf`
- `text/plain`

Maximum attachment size is 10 MB.

## 7. CSV Import/Export

CSV import/export uses Apache Commons CSV.

The CSV columns are:

```text
id,title,description,status,priority,type,assigneeId
```

Export behavior:

- Exports non-deleted tickets for the requested project.
- Uses Apache Commons CSV so commas, quotes, and newlines are escaped correctly.
- Does not include `dueDate`, because the README/PDF CSV contract does not include it.

Import behavior:

- `projectId` comes from the multipart form field, not from the CSV.
- The CSV `id` column is ignored because database IDs are generated by the backend.
- Bad rows are counted as failures and reported with row-level errors.
- A bad row does not abort the whole import.
- Valid rows still create tickets through the normal ticket creation service, preserving validation, explicit assignee checks, auto-assignment, and audit logging.

## 8. Auto-Escalation Scheduler

Automatic overdue escalation uses Spring `@Scheduled`.

The scheduler evaluates tickets that are:

- Not deleted
- Not `DONE`
- Assigned a `dueDate`
- Past their `dueDate`

Escalation behavior:

- `LOW -> MEDIUM`
- `MEDIUM -> HIGH`
- `HIGH -> CRITICAL`
- `CRITICAL` remains `CRITICAL`
- If a ticket is `CRITICAL` and still overdue, `isOverdue` is set to `true`
- Ticket status is not changed by escalation

Automatic escalation records `AUTO_ESCALATE` audit logs with actor `SYSTEM` only when a ticket actually changes.

## 9. Audit Log Coverage

The system records audit logs for implemented state-changing actions.

Read-only endpoints are intentionally not audited.

`POST /auth/login` is not audited because it does not change domain/application state. `POST /auth/logout` is audited because it persists a revoked token.

Audited user/auth actions:

- `CREATE_USER`
- `UPDATE_USER`
- `DELETE_USER`
- `LOGOUT`

Audited project actions:

- `CREATE_PROJECT`
- `UPDATE_PROJECT`
- `DELETE_PROJECT`
- `RESTORE_PROJECT`

Audited ticket actions:

- `CREATE_TICKET`
- `UPDATE_TICKET`
- `DELETE_TICKET`
- `RESTORE_TICKET`

Audited comment actions:

- `ADD_COMMENT`
- `UPDATE_COMMENT`
- `DELETE_COMMENT`

Audited dependency actions:

- `ADD_DEPENDENCY`
- `REMOVE_DEPENDENCY`

Audited attachment actions:

- `UPLOAD_ATTACHMENT`
- `DELETE_ATTACHMENT`

Audited CSV/system actions:

- `IMPORT`
- `AUTO_ASSIGN`
- `AUTO_ESCALATE`

The read-only Audit Log API is available at:

```http
GET /audit-logs
```

It supports optional filters:

- `entityType`
- `entityId`
- `action`
- `actor`
